下面这份是 **Fabric 1.21.4 服务端性能优化指南**，按"装这些就行 → 还掉 tps 怎么排查"的顺序来。

---

## 一、必装核心性能 mod（5 个，全部装无脑收益）

| Mod | 作用 | 行为是否变化 | 备注 |
|---|---|---|---|
| **Lithium** | 重写 entity tick / AI / 物品堆叠 / 红石计算等核心游戏逻辑 | ❌ 完全 vanilla 行为 | 性能 mod 之王，30%+ TPS 提升常见 |
| **FerriteCore** | 大幅压缩 blockstate / model 内存占用 | ❌ | 内存占用减少 30-50%，间接降 GC 压力 |
| **Krypton** | 优化 Netty 网络栈 + 数据包压缩 | ❌ | 减少网络包打包/拆包 CPU 开销，玩家多时收益明显 |
| **Moonrise** | Spottedleaf 写的"光照引擎 + 区块系统"重写（≈Starlight 升级版） | ❌ | **强烈推荐**，1.21+ 取代了原 Starlight，光照刷新速度提升数倍 |
| **ServerCore** | 一堆**可配置**的服务端优化（spawning、chunk tick、entity collide 等） | ⚠️ 默认配置不改行为；激进配置会改 | 装上后**先用默认配置跑一段**，按需开 |

下载地址都在 [Modrinth](https://modrinth.com/mods?f=categories%3Aperformance&g=categories%3Afabric)，搜名字就能找到。

> **重要**：Moonrise 装上之后**不要再装 Starlight**（功能被 Moonrise 包含了），两个一起会冲突。

## 二、高负载场景的进阶 mod（按需装）

只在"玩家多 / 区块多 / 实体多"时才需要：

| Mod | 适用场景 | 注意事项 |
|---|---|---|
| **C2ME**（Concurrent Chunk Management Engine） | 多人 / 大世界 / 频繁 chunk load | 多线程 chunk I/O，**和 Moonrise 兼容**但要看版本对应表 |
| **VMP**（Very Many Players） | 在线 ≥ 20 玩家 | 优化玩家追踪 / 实体追踪广播；CTT 这种小队场景**通常不用** |
| **AsyncPathing** / **AsyncPlayerPathing** | 怪多、AI 卡 | 把寻路移到异步线程；地图自定义 AI 多时收益大 |
| **Alternate Current** | 红石线路重 | 重写红石更新算法；CTT datapack 不依赖红石就**不用装** |
| **MemoryLeakFix** | 长时间运行 RAM 涨不停 | 修复 vanilla 已知内存泄漏；长在线推荐装 |

对 **CTT 服务器**来说，第一档 5 个 + `MemoryLeakFix` 基本就够了。`C2ME` / `VMP` 看在线人数决定。

## 三、必装诊断工具（不装就是瞎调）

| Mod | 作用 |
|---|---|
| **Spark**（[Modrinth 链接](https://modrinth.com/mod/spark)） | **必装**。CPU profiler + 内存 dump + tickmonitor。掉 TPS 时第一件事就是 `/spark profiler --timeout 60`，会输出火焰图给你看到底是哪个 mod / vanilla 子系统在吃 CPU |
| **Observable**（可选） | 服务端可视化 TPS / entity heatmap。在游戏里直接画热力图，找到"是哪个区块在拖" |

**没有 Spark 等于在猜**，先装它再调任何东西。

## 四、配置调优（不改 mod 也能省一半 CPU）

### 4.1 `server.properties`

```
view-distance=8           # 默认 10，公服建议 6-8
simulation-distance=6     # 默认 10，建议 4-6（CTT 这种关卡式地图甚至 4 都够）
max-tick-time=-1          # 关掉 watchdog 误杀（崩服时方便看完整 stack）
```

`simulation-distance` 是 **TPS 杀手最大变量**——它决定了实体 tick / 红石 tick / 怪物刷新的范围。CTT 关卡是固定的小区域，**没必要 simulate 远处区块**，调到 4-5 即可。

### 4.2 `ServerCore` 关键配置（`config/servercore.json`）

激活后逐项改，每改一项观察 `/spark tps`：

```jsonc
{
  "optimizations": {
    "disable_spawn_chunks": true,         // 关掉 spawn chunk 永久 tick（vanilla 默认会浪费一片）
    "async_chunk_loading": true,
    "merge_nearby_items": true,           // 掉落物聚合，物品多时省 entity tick
    "dont_load_worldborder": true
  },
  "mob_spawning": {
    "enabled": true,
    "interval": 2                          // 怪物刷新检查间隔从 1 tick 提到 2-4 tick
  },
  "experience_merging": { "enabled": true } // 经验球聚合
}
```

### 4.3 view-distance / mob caps（看人数）

| 在线人数 | view-distance | simulation-distance | mob-cap |
|---|---|---|---|
| ≤ 5 | 8-10 | 6-8 | vanilla 默认 |
| 6-15 | 6-8 | 4-6 | vanilla 默认 |
| 16-30 | 5-6 | 4 | 适当降低 |
| > 30 | 4-5 | 3-4 | 必须降低 + 装 VMP |

## 五、JVM 参数（很多人忽略，影响巨大）

### 5.1 推荐 JVM args（Java 21 + 8-16GB heap）

```bash
java \
  -Xms8G -Xmx8G \
  -XX:+UseG1GC \
  -XX:+ParallelRefProcEnabled \
  -XX:MaxGCPauseMillis=200 \
  -XX:+UnlockExperimentalVMOptions \
  -XX:+DisableExplicitGC \
  -XX:+AlwaysPreTouch \
  -XX:G1NewSizePercent=30 \
  -XX:G1MaxNewSizePercent=40 \
  -XX:G1HeapRegionSize=8M \
  -XX:G1ReservePercent=20 \
  -XX:G1HeapWastePercent=5 \
  -XX:G1MixedGCCountTarget=4 \
  -XX:InitiatingHeapOccupancyPercent=15 \
  -XX:G1MixedGCLiveThresholdPercent=90 \
  -XX:G1RSetUpdatingPauseTimePercent=5 \
  -XX:SurvivorRatio=32 \
  -XX:+PerfDisableSharedMem \
  -XX:MaxTenuringThreshold=1 \
  -jar fabric-server-launch.jar nogui
```

这套是 Aikar 调过的 G1GC 参数，**对 Minecraft 服务器特别友好**——重点是减少 GC 暂停时间，避免 GC 引起的瞬时 TPS 跳水。

### 5.2 关键点

- **`-Xms` 和 `-Xmx` 必须相同**（避免堆动态扩容卡顿）。
- **不要分配过多内存**：32GB 物理机给 JVM 16GB 够用，留一半给操作系统 / 文件缓存。给 24GB 反而会让 GC 暂停变长。
- **Java 21**（你 build.gradle 里已经是 Java 21 了，Fabric 1.21.4 强制要求）。
- **如果 GC 是瓶颈**，可以试 ZGC（`-XX:+UseZGC -XX:+ZGenerational`），暂停时间 < 1ms，但吞吐略低。32GB+ heap 推荐 ZGC，16GB 以下还是 G1GC。

## 六、TPS 掉的排查流程（实战版）

按这个顺序走，**每一步只改一件事**，方便定位：

### Step 1 · 抓 profiler 看大头

```
/spark profiler --timeout 60
```

等 60 秒后看输出 URL，看 CPU 时间占比最高的子系统：
- **`ServerLevel.tick` > 70%** → 实体 / 红石 / chunk tick 问题，去看 4.1 的 simulation-distance；
- **`ChunkMap.tick` 或 `ChunkLoad` 高** → 区块加载问题，装 C2ME；
- **`ServerLevel.tickPath` 或 `Mob#tick` 高** → 怪物 AI / 寻路，装 AsyncPathing；
- **`Scoreboard.updateScore` 或 `text_display` 高** → datapack 写计分板太密集，**这是 CTT 地图常见问题**，下面 §6.5 单独讲。

### Step 2 · 看是不是某个 mod 在拖

profiler 输出里会按 mod 包名分组。如果看到某个 mod 占 > 5% CPU 但你又用不上（典型例子：经济 / 商店 / 大世界生成 mod 但你的服根本不需要），直接卸了。

### Step 3 · `/spark tickmonitor` 实时盯 lag spike

```
/spark tickmonitor --threshold 100
```

任何超过 100ms 的 tick 会立刻打印 stack trace，**特别适合抓"偶尔卡一下"的间歇性掉 tps**——这种 profiler 平均化反而看不出来。

### Step 4 · 看内存压力

```
/spark heapsummary
```

如果 `RAM used > 80%` 持续不降，是 GC 频繁→**先加内存或装 FerriteCore + MemoryLeakFix**。

### Step 5 · CTT 地图特有的"计分板风暴"问题

CTT 这类地图每 tick 写几百次计分板（每只怪都有 Marker / *DMG / RedHearts）。如果 profiler 看到 `Scoreboard.updateScore` / `ScoreboardScoreUpdateS2CPacket` 占比 > 10%，说明被它拖了。**这种问题不能靠性能 mod 解决**，只能：

1. 让地图作者把高频写入的 objective 数据合并；
2. 或者卸掉读取这些计分板的 mod / 插件；
3. 或者降 simulation-distance 让远处的怪不被 tick → 间接减少计分板写入量。

## 七、推荐组合（直接给你朋友用）

**给"不想折腾、最大化收益"的标准包**（约 10 个 mod）：

```
必装：
- Lithium
- FerriteCore
- Krypton
- Moonrise
- ServerCore           （先用默认配置）
- MemoryLeakFix
- Spark                （诊断必装）

可选（看负载）：
- C2ME                 （在线 > 10 / 大世界）
- VMP                  （在线 > 20）
- AsyncPathing         （怪 AI 多）
```

**安装步骤**：

1. 进 [Modrinth](https://modrinth.com)，每个 mod 选 **Fabric** + **1.21.4** + **Server** 标签下载；
2. 同时下载它们的依赖（`Fabric API` 必须装；ServerCore 等可能需要别的库 mod）；
3. 全部丢到服务器 `mods/` 文件夹；
4. 启动一次，看启动日志有没有红字（缺依赖会提示）；
5. 用 `/spark profiler --timeout 60` 跑一次基线，对比装前后 TPS。

## 八、不要装的"反优化"mod（避坑）

下面这些常见但**不推荐装**：

| Mod | 原因 |
|---|---|
| **Phosphor** | 已被 Starlight / Moonrise 取代，1.21 没维护 |
| **Sodium / Iris** | 客户端 mod，服务端装了无效 |
| **ClumpFabric** | 经验球聚合，**ServerCore 已经包含同功能**，重复装会冲突 |
| **Spark Profiler 老版本（< 1.10）** | 1.21 兼容性差，必须用最新版 |
| **Carpet（不开 ML 模式）** | 是地图制作工具，**不是性能 mod**；不开发就别装 |
