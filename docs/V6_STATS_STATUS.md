## §2 · 四项指标 · 逐项对照

原设计：造伤 / 承伤 / 击杀 / 治疗，四个指标。

| 指标 | 字段 | 设计来源 | 当前实现 | 状态 |
|---|---|---|---|---|
| **造伤** | `dmg_dealt` | 监听 `Damage` scoreboard + 四层血 delta 校验 | `PlayerDamageStats.confirmed`（按玩家累积；来源于 `*DMG` delta + `DamageShower` 文本实体） | ⚠ **路径不同**：原设计要以 `Damage` scoreboard + 四层血 delta 校验，现实是分 9 种 `*DMG`；数值结果应一致，但溯源能力（按元素拆分）是原设计没有的额外维度 |
| **承伤** | `dmg_taken` | 同造伤但 target 筛 CTT 玩家 | **无任何相关采集代码**（grep `dmg_taken` / `dmgTaken` 均 0 命中） | ❌ |
| **击杀** | `kills_mob` | `RedHearts` 从 `>0` 跌到 `≤0` | **无任何相关采集代码**（grep `kill` / `Kill` / `击杀` 均 0 命中） | ❌ |
| **治疗** | `heals_teammate` | Mixin 主动治疗 + 目标玩家正 delta | **无**（v3 才做，当前仍是 v6.3.10） | ❌ 按设计节奏尚未到 v6.3.x 该做的项 |

**额外发现**：

- **数值定义**（设计 §2 铁律）："最终落到敌方血量上的数字、永远非负、下限 1"——当前实现满足（`*DMG` delta 天然非负，`DamageShower` 有 clamp）
- **反射伤害 / 环境伤害**（设计：落到 `unattributed`）——当前实现 ✅：`AttackerProbe` 的 L9_NONE 恰好承担此职责
- **归属策略**（设计：5 tick 滑窗 UUID 匹配）——当前实现 ➕ **严重超规**：实际做了九层 L1~L9 + L7b 栈，复杂度远超原设计。这不是坏事（准确率更高），但**说明开发过程发现"5 tick 滑窗"根本不够用**

## §3 · 统计范围与生命周期 · 全面缺失

原设计的三个概念：**Session**（一局，由 GameID 分界）、**Stage**（一关，由 6 个 stage holder 分界）、**大厅/小游戏过滤**。

| 设计项 | 触发条件 | 当前实现 | 状态 |
|---|---|---|---|
| Stage 开始 | 6 个 stage holder 之一由 0 跳到非 0 | 无监听 | ❌ |
| Stage 结束 | 6 个 stage holder 全回到 0 | 无监听 | ❌ |
| Stage 唯一键 `(GameID, Tier, Floor, StageType, StageNum)` | 五元组聚合键 | 无概念 | ❌ |
| 检查点 `#CheckPoint=1` 不清零 | — | 无监听 | ❌ |
| 休息室暂停采集 | — | 无；**当前是 L 键手动 start/clear/freeze/stop** | 🔄 改为手动会话 |
| Session 开始 | `#CTT GameID` 跳变 | 无监听 | ❌ |
| Session 落盘 `world/data/ctt_stats.dat` | 每 30s + 每关结束 | **完全无持久化**；重启游戏数据归零 | ❌ |
| `isInLobby()` / `#LobbyMiniGame > 0` 过滤 | — | 无判定；当前所有环境都采集 | ❌ |
| 休息室假人天然过滤 | 依赖"休息室不采集" | 由于没有休息室判定，**假人伤害会被记进面板** | ❌ 潜在误差源 |

**替代品**：当前 `PlayerDamageStats.start/clear/freeze/stop` 提供了一个**手动的会话**，语义上靠近 "session" 但**完全由玩家按钮控制**，与 GameID 跳变无关。这等于把生命周期决定权推给了用户，功能还在，但自动化和回档一致性没了。

## §4 · 数据聚合维度 · 维度缺失

原设计要求同一事件落入**三个桶**：

| 桶 | 设计用途 | 当前实现 | 状态 |
|---|---|---|---|
| `player × stage` | 分关表每行 | 无 | ❌ |
| `player × session` | 嵌入 HUD 下半行 + 总表 | 有（`PlayerDamageStats.Entry` 单层） | ⚠ 单桶而非二维 |
| `team × stage` | 单关总量 / 贡献比例 | 无 | ❌ |

当前实现相当于**只有 "player × 当前会话"** 一个维度，跟设计 1/3 匹配。

## §5 · 嵌入式 HUD · 完全未做

原设计：每个队友 3 行（血量行 + `关: ⚔ ⛨ ☠` + `局: ⚔ ⛨ ☠`），Unicode 图标 / 数字压缩 / 自己金色 / 离线灰。

| 设计项 | 当前实现 | 状态 |
|---|---|---|
| 队友面板扩两行 | 队友面板仍是 v5.3.3 单行（血条 + 命数） | ❌ |
| `⚔ ⛨ ☠` Unicode 图标 | 未接入 | ❌ |
| 数字格式化 `12.3k / 1.2M` | `DamagePanelRenderer` 有本地 `fmt()` 做 `k/M` 压缩，**但只在伤害分配面板里用**，未下放到队友行 | ⚠ 能力已在代码里但没复用 |
| 自己金色 / 队友白 / 离线灰 | 队友面板沿用 v5.3.3 原色；**离线玩家根本没有概念** | ❌ |
| 休息室显示"上一关"、大厅隐身 | 无 Stage 概念 → 无对应行为 | ❌ |

**结论**：整个 §5 章节 0% 交付。

## §6 · 表格面板（K 键） vs 实现的"伤害分配面板"（L 键）

这是偏离最大的一节——做出来了一个面板，但几乎是另一个东西。

### 6.1 形态差异

| 维度 | 设计 §6（K 键） | 当前实现（L 键） |
|---|---|---|
| 触发键 | `K` | `L`（`K` 被 `toggle_bossbars` 占用） |
| 切换 | toggle，再按关闭 | toggle，再按关闭 ✅ |
| 遮罩 | 半透明灰覆盖层（Tab 风） | **透明无灰底**（v6.3.1 用户明确要求去掉） |
| 游戏是否暂停 | 否 | 否 ✅ |
| 关闭 | `K` / `Esc` / 开背包 | `L` / `Esc`（开背包会自然失焦） ✅ 近似 |
| 顶部栏 | 当前关卡标签 + 总表/分关表切换 + Session 计时 | 无顶栏；只有时长行 |
| 标签页 | 两个（总表 / 分关表） | **无标签页**；单视图 |

### 6.2 总表（标签页 1）· 列结构对照

| 列 | 设计 §6.4 | 当前 `DamagePanelRenderer` | 状态 |
|---|---|---|---|
| 头像 | 皮肤 8×8 | 无 | ❌ |
| 玩家名 | 文字（自己金 / 离线灰带`[离线]`） | 文字，无离线状态 | ⚠ |
| ⚔ 造伤 | 有 | 有（`confirmed`） | ✅ |
| ⛨ 承伤 | 有 | 无 | ❌ |
| ☠ 击杀 | 有 | 无 | ❌ |
| ⏱ 在线时长 | 有 | 无（只有面板会话总时长） | ❌ |
| ♥ 治疗（v3） | 有 | 无 | ❌ 按设计节奏暂不该做 |
| **排序** | 默认 ⚔ 降序；列头点击切换 | ⚔ 降序固定，**不支持列头排序** | ⚠ |
| **自己行高亮** | 淡金背景 + 金名 | 名字不区分 | ❌ |
| **离线玩家** | 50% 透明 + `[离线]` 后缀 | 无概念 | ❌ |

### 6.3 分关表（标签页 2）· 0% 实现

整个分关表（Stage 分组、按关卡层数升序、同关内 ⚔ 降序、当前关描金边）**完全没有**，因为没有 Stage 概念（见 §3）。

### 6.4 滚动 / 空状态 / 颜色

| 设计项 | 当前实现 | 状态 |
|---|---|---|
| 行数溢出显示滚动条（右侧 + 滚轮 + 拖拽） | 行数直接溢出画面，无滚动条 | ❌ |
| 当前关卡自动 keep-alive | 无关卡概念 | ❌ |
| 空状态占位（"尚无数据"） | 空会话时面板显示全 0 | ⚠ 功能上 OK 但无设计中的装饰提示 |
| 配色 `#181818 背景 / #FFAA00 自己 / #888888 离线` | 透明底 + 纯文字 | 🔄 风格改了 |

### 6.5 设计外新增 ➕

| 实现里多出来的东西 | 设计中是否存在 |
|---|---|
| 面板内四按钮 ▶ 暂停 / ✕ 清空 / ⚑ 快照 / ⚙ 设置 | ❌ 设计没有 |
| 拖拽标题栏移动面板 | ❌ 设计没有（设计是固定位置 Tab 风） |
| 玩家行详情：事件数 / 最高单次 / 各层命中计数 | ❌ 设计没有 |
| 未分类伤害桶单行 | ⚠ 设计 §9 有 `showUnattributed` 配置，但当前是**常驻显示**而非可选 |

## 建议（仅陈述事实，不作计划；计划见 `ROADMAP.md`）

从"对齐原设计"的视角看，下列事项是 gap 中**规模最大、基础性最强**的三条：

1. **Session / Stage 生命周期**（§3）——是 §4/§5/§6 几乎所有后续功能的前置
2. **持久化 `ctt_stats.dat`**（§3.2 / §8.5）——没有它，session 概念没意义
3. **另外三个指标**（承伤 / 击杀 / 治疗）——现在只差 3/4 的数据采集

`ROADMAP.md §6 待办`处目前列的"击杀统计 + 异常过滤器"只覆盖了上述 gap 的一部分，后续可据本文档扩充待办清单。
