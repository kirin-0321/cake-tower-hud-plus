# 异常问题记录

## 2026-05-01 玩家反馈

### 1. 伤害统计数值异常飙高 ✅ 已修复（v8.3.2 · 2026-05-01）
- **场景**：地图 "Sword fights on the heights"（高空剑斗）等含脚本"机制斩杀 / 假伤害"的关卡
- **现象**：所有玩家的 "Damage Dealt"（造成伤害）远超正常值，每人显示约 200K（20 万）伤害
- **根因**：地图 datapack 用 `scoreboard players set/remove ... RedHearts <大数>` 直接刷"假伤害"——
  常见值 1000 / 9000 / 10000 / 100000；这些数被本模组的 `RedHearts↓ → DamageProbe → AttackerProbe`
  归属链当作真实伤害归到附近玩家头上，多个 victim × 多 tick 持续刷入后累计虚高
- **修复（v8.3.2 · G7b 三指标 AND + 致死 capped 补偿）**：
  - 新增 `server/filter/PendingDamageBuffer.java`：放行事件先入队 delay 2 tick 再判定，
    捕获入队瞬间 `attackerUuid / weaponId / wasLethalAtBuffer`
  - 新增 `server/filter/PerPlayerWeaponP95Registry.java`：`Map<UUID, Map<weaponId, RollingWindow>>`
    per-(player, weapon) 双层桶，`RollingWindow.java` N=100 二分插入排序 O(log N) 查 P95
  - 新增 `server/filter/WeaponIdResolver.java`：`custom_data` key 优先 + vanilla item id
    兜底，空手返回 `"empty"`
  - 改 `PlayerDpsTracker`：新增 `dpsActive(uuid)` + `dpsActiveByWeapon(uuid, weaponId)`，
    返回最近 5 秒**非空桶平均**（避免被空闲秒稀释，反映真实出手节奏）
  - 改 `DamageFilterPipeline.evaluateOutlierG7b`：**三道门 AND**——
    `damage > outlierAbsoluteFloor (=800)` ∧ `damage > P95 × K_p (=3)` ∧ `damage > DPS_active × M_d (=1.5)`
    全过才标 `outlier` 或 `lethal-mechanism`，否则放行；样本不足 / 归属失败任一门跳过自然降级
  - 改 `DamageProbe.recordFromRedHearts`：同步规则栈未命中事件改入 buffer；`flushTick`
    末尾批量 drain 出来跑 G7b 判定；命中即路由 `L9_FILTER` + reason tag
  - **入样规则**（防自污染）：被任何过滤路径处理的事件**不**写入 P95 / DPS 训练样本——
    避免"被过滤的大数把自身阈值拉高 → 下次同样的大数反而过不掉"的恶性循环
  - **致死 capped 补偿**（`DamageProbe.recordLethalCappedDamage`）：玩家用秒杀技 / 超额武器
    （如 9999 dmg）击杀小怪（MaxHP=200）触发 `lethal-mechanism` 时，把
    `min(damage, MaxHP)` 加到玩家伤害账户——避免击杀维度归属对了而伤害维度归零的体验割裂
  - 改 `ServerConfig.java`（version 4→5）：新增 9 个调参字段（`useDamageBuffer / bufferDelayTicks / bufferMaxSize /
    p95WindowSize / p95MinSamples / p95OutlierMultiplier / dpsActiveMultiplier / outlierAbsoluteFloor /
    p95OutlierEnabled`）
  - 改 `StatsSnapshotPayload` 升 v3 schema：PlayerEntry 末尾追加
    `currentWeaponId / weaponDpsActive / weaponP95 / weaponP95Samples` 4 字段
  - 改 `DamagePanelRenderer.drawWeaponG7b`：伤害分配面板顶部新增一行实时显示
    `武器 X  P95=N(s/cap)×3=t1  DPS=D×1.5=t2  fl=800`，方便玩家观察当前主手武器三道门状态
  - 改 `PlayerDamageStats.start/stop/clear` + `CttStatsServer.DISCONNECT`：
    联动清空 P95 / buffer / per-weapon DPS 桶，避免跨局污染 / 内存泄漏
- **设计文档**：[大额伤害过滤器.md](大额伤害过滤器.md)（白话版 · 三道门类比）
- **回落兼容**：`useDamageBuffer = false` → 退化为 v6.7.x 同步规则栈直接归属；
  `outlierAbsoluteFloor = 0` → 降级为 V2.1 双指标 AND（floor 门关闭）
- **关键参数对照**：合法最大单刀 600（C 武器），异常 1000——`floor=800` 在中间画一条数学红线，
  保证合法 ≤ 600 永远放行 + 异常 1000 永远被三门 AND 命中拦下；具体调参指南见
  [大额伤害过滤器.md](大额伤害过滤器.md) §八
- **相关文件**：`src/main/java/com/ctt/healthdisplay/{server/filter/*, server/DamageProbe.java, server/PlayerDpsTracker.java,
  config/ServerConfig.java, network/StatsSnapshotPayload.java, hud/DamagePanelRenderer.java}`

### 2. /trigger ViewStats 命令刷屏 ✅ 已修复（方案一 · 2026-05-01）
- **现象**：玩家不断在聊天框输入 `/trigger ViewStats`（推测是为了触发左上角的统计信息显示）
- **问题**：OP 权限玩家能看到所有人输入的这个命令，导致聊天框被严重刷屏
- **诉求**：希望能让该命令的执行/输出对其他玩家（尤其是 OP）不可见
- **修复内容**：
  - 新增 `client/TriggerEchoFilter.java`，在客户端 `ClientReceiveMessageEvents.ALLOW_GAME`
    事件里**无条件屏蔽**本模组自发 trigger 命令的聊天回显
  - 覆盖命令：`ViewStats`、`TogglePartyBossbar`
  - 覆盖消息形态：
    - 自己手动 / 自动触发的 `Triggered [XXX]` 成功回显
    - OP 看到的 admin 广播 `[PlayerName: Triggered [XXX]]`
    - 中文本地化 `已触发 [XXX]`
  - 仅影响本地渲染，不改服务端记账；未来要禁言的命令名在 `TriggerEchoFilter.MUTED_TRIGGERS` 追加即可
- **未做的方案二（后续可选）**：从根源上使用服务端发送属性数据的方式，
  彻底省掉客户端主动发 `/trigger ViewStats` 的需要。

### 3. 队友 Boss 血条闪烁 + 队友 HUD 闪烁 ✅ 已修复（2026-05-01）
- **场景**：breakroom（休息室）及其他有队伍 bossbar 的场景
- **现象**：队友 bossbar 不停闪烁，**连带**本模组的队友血量 HUD 也闪烁；关闭 `hideTeamBar` 就不闪
- **根因**：`BossBarHudRenderMixin` 用"HEAD 移除 / RETURN 塞回" 策略隐藏指定 bar，副作用严重：
  1. `LinkedHashMap` 插入顺序被打乱（remove + put → bar 永远被推到末尾）
  2. `render` 方法首行 `if (bossBars.isEmpty()) return;`：若当前只剩 teamBar 一条，移除后 Map 变空 → vanilla early return → Profiler 都没 push；下一帧再重复 → 状态每帧抖动
  3. **最关键**：其他代码（包括本模组 `HealthData.parseBossBarData` 每 tick 读一次）如果恰好在 HEAD 与 RETURN 之间访问 `bossBars`，会看到"空 Map" → `hasTeamBossBar=false` → `teammates.clear()` → 队友 HUD 内容被清空一帧 → 跟着 boss bar 一起闪
- **修复**：改用 Mixin `@Redirect` 拦截 `render` 方法里对 `bossBars.values()` 的调用，返回过滤掉隐藏 UUID 的快照 List。**原 Map 永远不被修改**，三个副作用全部消除。
  - 基于 1.21.4 yarn 1.21.4+build.8 的字节码验证过 target（`javap -c` 确认 render 里用的是 `values()`）
  - 每帧一次 O(n) 额外分配，n ≤ 5（bossbar 数量），开销可忽略
- **相关文件**：`src/main/java/com/ctt/healthdisplay/mixin/BossBarHudRenderMixin.java`

### 4. 存在 Boss 冠军时敌人头顶血条错乱 ✅ 已修复（M7 · v8.3.0 · 2026-05-01）
- **场景**：Towers 模式下有 Boss 级别冠军怪存在时
- **现象**：敌人头顶 HP 血条显示出现 glitch/错乱
- **根因**：vanilla bossbar 同一时刻只会出现一条（Boss 出现后永远锁 Boss），客户端
  旧路径 `CttHealthDisplay.updateMobTracking` 只能通过"把 bossbar 里的 HP 贴给
  最近同名 mob"的近似搜索来给每只怪估血量。Boss 冠军存在时 bossbar 里的数值是
  **Boss 自己** 的 HP，被错误地复制给了近身的普通精英 / 召唤物 → 头顶条大面积错位。
- **修复（M7 · 服务端权威怪物血量管道）**：
  - 新增 `network/MobHealthPayload.java`（UUID / name / hp / maxHp / targetted / nameColor）
  - 新增 `server/MobHealthBroadcaster.java`：`END_SERVER_TICK` 每 5 tick，按在线玩家
    48 格视野扫活 LivingEntity，读 `RedHearts` / `MaxHP` scoreboard，按距离排序
    取前 32 条差量推送；最近那只标 `targetted=true`
  - 新增 `server/ScoreboardReader.java`：null 防御的 objective + score 读取小工具
  - 新增 `client/ClientMobHealthCache.java`：接包构造 `MobHealthData` snapshot，
    `isFresh(5 s)` 视作权威数据可用
  - 改 `health/HealthData.getMobHealthMap()`：顶部分叉 —— fresh 返回服务端 snapshot，
    否则回落本地 `mobHealthMap`
  - 改 `CttHealthDisplay.updateMobTracking`：fresh 时整段客户端近似链路短路
  - 改 `CttStatsServer / CttHealthDisplay`：双端注册 `MobHealthPayload` + DISCONNECT reset
- **回落兼容**：服务端没装本 mod → 客户端永远收不到包 → `isFresh()` 恒 false →
  行为与 v8.2 完全一致，老玩家零感知
- **相关文件**：`src/main/java/com/ctt/healthdisplay/{network,server,client,health}` 全家桶

### 5. 服务器 `[@] L` 单字噪声刷屏 ✅ 已修复（v8.3.5 · 2026-05-01）
- **现象**：服务器（地图脚本 / datapack / tellraw）主动 broadcast 形如 `[@] L` 的单字提示，
  几秒就刷一屏；本地装的"聊天合并"类 mod 进一步折叠成 `[@] L (x137)`，正常聊天被淹没
- **诉求**：客户端无条件抹掉这种固定字面噪声，不影响服务端记账
- **三步迭代踩坑**：
  - **v8.3.3 初版（漏）**：单纯 `body.equals("[@] L")`，只能命中服务端 `tellraw` 直接发裸
    文本的情况；漏掉 announcement / chat.type.text / admin 广播三种 translate 壳
  - **v8.3.4 字符串兜底（仍漏）**：补上 `endsWith(" " + muted)`、`endsWith(": " + muted + "]")`
    覆盖玩家发话与 admin 广播；但被合并 mod / 时间戳 mod 在 `Text` 树前后塞 hover 装饰节点
    时，`getString()` 拼接后可能多出隐形空白，equals/endsWith 仍偶尔漏。**关键证据**：用户
    抓到的实际包是 `chat.type.announcement` 模板（不是 admin 广播），结构 ≈
    `{extra: [{text:"", hoverEvent:..时间戳..}, {translate:"chat.type.announcement", with:["@","L"]}, {text:" (x181)"}]}`
  - **v8.3.5 结构化匹配（最终版）**：递归遍历 `Text` 树找 `TranslatableTextContent` 节点，
    直接读 `getKey()` + `getArgs()`，比对 `(sender, message)` 二元组——**完全绕开渲染层**，
    不受时间戳 mod 装饰、不受合并 mod 后缀、不受 zh/en 翻译差异影响。字符串 endsWith 路径
    保留作兜底（处理服务端用 `tellraw` 发裸文本的情况）
- **黑名单维护**：
  - 主路径精准匹配（推荐）：`ServerChatNoiseFilter.MUTED_ANNOUNCEMENTS`，`{sender, message}` 数组，
    配合 `CHAT_TRANSLATE_KEYS`（announcement / text / admin 三种壳共用），命中需 sender + message
    都吻合，绝不误杀正常 "L" 聊天
  - 字符串兜底（备用）：`ServerChatNoiseFilter.MUTED_LINES`，含整体形态 `"[@] L"`
- **挂载点**：`CttHealthDisplay.ALLOW_GAME` 里紧跟 `TriggerEchoFilter.shouldHide` 后调用，
  命中即 `return false` 阻止入聊天 HUD（不影响服务端记账）
- **相关文件**：`src/main/java/com/ctt/healthdisplay/client/ServerChatNoiseFilter.java`、
  `src/main/java/com/ctt/healthdisplay/CttHealthDisplay.java`
